# Plan: Improve Context Window Management (Task 3b327411)

## Objective
Implement a sophisticated "memory" system that moves beyond simple sliding windows or blanket summarization. The goal is to identify and retain high-value facts while discarding or summarizing less relevant information to maximize the utility of the available context window.

## Current Limitations
- **Simple Sliding Window:** Older information is lost regardless of its importance.
- **Naive Summarization:** All past turns are compressed equally, potentially losing specific details (e.g., a specific variable name or a unique constraint) that are crucial for later steps.

## Proposed Solution Architecture

### 1. Fact Extraction & Categorization
Instead of treating the history as a raw string of messages, we will process recent interactions to extract "Facts".
- **Fact:** A discrete piece of information (e.g., "The user wants to use Python 3.10", "The project name is 'Alpha'", "The database schema requires a primary key on 'id'").
- **Metadata:** Each fact will have a timestamp and a relevance score.

### 2. Relevance Scoring System
A scoring algorithm will determine what stays in the "active" context:
- **Recency:** Higher weight for recent interactions.
- **Frequency:** Facts mentioned multiple times get higher scores.
- **Entity Recognition:** Specific entities (filenames, variable names, specific requirements) are flagged as high priority.
- **User Intent:** Explicit instructions from the user ("Remember that...", "Always use...") get maximum priority.

### 3. Dynamic Context Construction
The context sent to the LLM will be constructed as follows:
1.  **Core Facts:** A list of high-scoring facts (preserved in original detail).
2.  **Recent History:** The last $N$ turns of conversation (to maintain flow and immediate context).
3.  **Summarized Background:** Low-priority information from the distant past, condensed into a brief summary.

## Implementation Steps

### Phase 1: Analysis & Data Modeling
- [ ] Analyze current `core.py` or relevant module handling history.
- [ ] Define the `Fact` data structure (ID, Content, Score, Category).
- [ ] Create a `MemoryManager` class to handle fact extraction and scoring.

### Phase 2: Scoring Engine Development
- [ ] Implement a basic keyword/entity extractor.
- [ ] Develop the scoring algorithm (Recency + Frequency + Importance).
- [ ] Implement a "Garbage Collection" routine that prunes low-score facts from the long-term memory store.

### Phase 3: Integration & Refinement
- [ ] Modify the context builder to inject the "Core Facts" into the prompt.
- [ ] Adjust the system prompt to instruct the LLM on how to interpret the "Facts" section vs. the "Recent History".
- [ ] Test with a long conversation (20+ turns) to ensure specific details from turn 3 are still accessible at turn 25.

## Success Criteria
- **Persistence:** Specific technical constraints mentioned early in a session remain available in the prompt even after many turns.
- **Efficiency:** The total token count remains within limits while maintaining high "information density."
- **Accuracy:** The LLM correctly prioritizes facts over general conversational filler.
